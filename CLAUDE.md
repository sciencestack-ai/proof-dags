# Instructions for Claude (and other AI agents) working in this repo

This repository is a curated registry of typed proof dependency graphs over the math arXiv. Each DAG was generated by Claude Opus 4.7 reading source LaTeX (fetched via the ScienceStack API) under the discipline below, with maintainer review. The discipline is what distinguishes this work from a typical "AI-generated knowledge graph" demo that falls apart on close reading.

**If you are extending the registry** (adding a new program, new papers, new edges, or modifying existing ones), follow these rules. They are not stylistic — every one of them exists because a specific past failure cost real audit work.

---

## Hard rules (never violate)

1. **Never fabricate arXiv IDs.** Before adding any paper to a `manifest.json` / `queue.json` / `bibliography`, verify the arXiv ID against its title via the arXiv API:
   ```bash
   curl -sG "https://export.arxiv.org/api/query" \
     --data-urlencode "search_query=ti:\"<title fragment>\" AND au:<author>" \
     --data-urlencode "max_results=3"
   ```
   LLM-recalled arXiv IDs are unreliable. Paper title → arXiv ID is one-way: never the reverse.

2. **Never fabricate node IDs.** Every edge target must be a real node in a real `papers/<id>.json`. For cross-DAG edges, verify the target node exists in the *target DAG's* paper file before adding the edge.

3. **Never hallucinate math content.** Node summaries paraphrase what the source paper *actually says*. Fetch node content via the ScienceStack CLI (`sciencestack --output json nodes <paper> --ids <id>`) and read it before writing a summary. Never invent constants, statements, or attributions.

4. **Never claim agreement with a Lean blueprint without per-node verification.** "This edge agrees with the blueprint" is a checkable claim; "this DAG is verified against the blueprint" requires building the label-by-label correspondence table. If you haven't done the table, say "structural correspondence, label-by-label table not yet built."

## Edge typing discipline

Use the controlled vocabulary in [`README.md`](./README.md). Three categories:

- **Load-bearing:** `depends_on`, `imports_result`, `provides_input_to`. Direct logical dependency — the source paper *uses* the target result. Every load-bearing edge must be backed by an explicit quote or a clear logical use; route to human review if uncertain.
- **Thematic:** `addresses_same_question`, `shared_mechanism`, `shared_tool`. Topical overlap *without* logical dependency.
- **Refinement / lineage:** `improves`, `extends`, `generalizes`, `re_derives`, `transliterates`, `removes_hypothesis`, `synthesizes`. The newer/stronger paper points to the older/weaker.

**Read every edge as English before committing it.** The format is subject–verb–object: `{from} {type} {to}`. Examples:

- `A improves B` — A is the improvement (from=newer, to=older).
- `A depends_on B` — A needs B (from=depender, to=dependee).
- `A provides_input_to B` — A's result is an input to B's argument.
- `A imports_result B` — A cites B's theorem verbatim as a baseline.

If the sentence reads as mathematical nonsense or temporally impossible ("Diamonds 2017 `depends_on` Fargues-Scholze 2021"), **flip the direction**. This is the single highest-yield sanity check.

**Default to thematic when uncertain.** Over-claiming dependency damages the registry's credibility on the negative-space queries it exists to serve. An honest `addresses_same_question` is better than a wrong `depends_on`.

## KaTeX rule (every summary and note)

The viewer at [sciencestack.ai/graph](https://sciencestack.ai/graph) renders summaries with KaTeX. **Every math expression in `summary`, edge `note`, and architecture `description` fields MUST be wrapped in `$...$` with proper LaTeX commands.**

Never write:
- Raw Unicode math: `≤`, `≥`, `≪`, `√`, `∞`, `θ`, `α`, `β`, `γ`, `δ`, `ε`, `λ`, `μ`, `π`, `σ`, `τ`, `φ`, `χ`, `ψ`, `ω`, `Δ`, `Λ`, `Σ`, `Φ`, `Ω`, `∫`, `∑`, `∏`, `⊂`, `∈`, `∼`, `≈`, `≠`, `→`, `↦`, `⟹`, `±`, `×`, `·`, `ℕ`, `ℤ`, `ℚ`, `ℝ`, `ℂ`, `𝔭`, `𝔓`
- Naked subscripts/superscripts: `H_1`, `M^k`, `p_n`, `M_{50}` outside `$...$`
- Naked function names: `liminf`, `log`, `sqrt`, `sup`, `inf` outside `$...$`

Always write:
- `$H_1$`, `$M_k$`, `$p_n$`, `$\le$`, `$\ge$`, `$\ll$`, `$\sqrt{x}$`, `$\theta$`, `$\varepsilon$`, `$\infty$`, `$\mathbb{Q}$`, `$\mathbb{Z}$`, `$\mathfrak{p}$`, `$\mathcal{O}_K$`
- `$\liminf_n d_n / \log p_n = 0$`, not `liminf_n $d_n$ / log $p_n$ = 0`
- JSON-escape backslashes: in source JSON, `\le` becomes `\\le`, `\\theta`, `\\sqrt{x}`, `\\mathbb{Q}`

Don't wrap (these are document references, not math):
- Node IDs (`thm:1.5`, `lem:4.7`, `prop:2.1`)
- Paper IDs (`2311.05762v3`, `math/0110009v3`)
- Section numbers ("Section 5", "§3.2")
- Acronyms used as labels: `EH`, `BV`, `MPZ`, `DHL`, `PFR` (only wrap when they appear inside a formula: `$\mathrm{EH}[\theta]$`)

## Integrity gate (every per-paper DAG must pass)

Before considering a per-paper file `papers/<id>.json` complete:

1. Every `from` and `to` in `edges` resolves to a real key in `nodes`.
2. No self-edges (`from == to`).
3. Every node in `critical_path` exists in `nodes`; consecutive pairs are connected by an edge in `edges`.
4. Every node appears in at least one layer of `proof_architecture`.
5. `python3 -m json.tool <file>` parses successfully.
6. Scan every `summary` and `note` for raw Unicode math or naked subscripts — fix per the KaTeX rule.
7. Spot-verify 2–3 numerical claims in the summaries against the source via `sciencestack nodes <paper> --ids <id>`. Catches both LaTeX bugs and content errors at the same cost.

Report passing the gate as: `<file>: nodes=N edges=M cp=K -> PASS`.

## Cross-paper edge schema (the upload format)

Cross-paper edges live in `<topic>/graph.json` and use **prefixed string IDs** for `from` / `to`:

```json
{
  "from": "2311.05762:thm:1.8",
  "to":   "2306.13403:prop:1.4",
  "type": "imports_result",
  "note": "..."
}
```

The full `from` value has *two colons*: one separating arxiv-id from node-id, and one inside the node-id (since the node-id itself is `thm:1.8`, `prop:1.4`, etc.). The upload script splits on the first `:` whose right-side prefix matches a known node-type prefix (`thm`, `prop`, `lem`, `cor`, `def`, `conjecture`, …).

**Do NOT use separate `from_paper` / `from_node` keys** — the upload script will silently skip cross-paper edges with that shape. Cross-check against `maynard-small-gaps/graph.json` or `prime-number-theorem/graph.json` before adding.

## When NOT to add a paper to a DAG

A paper belongs in a DAG only if it has **a typed edge into a node of the program** (or into a program-foundational paper). "Cites a paper in the program" is not enough — that's citation-graph noise, not proof-dependency structure.

If a sweep brings in papers that have no typed edge into the program's neighborhood, they should be parked, not added. The `pfr/_parked/` pattern (with a `NOTE.md` explaining why each was rejected) is the local-scratch convention — gitignored from the public dataset.

## Granularity

This is a **navigation graph at theorem/proposition granularity**, not a formalization blueprint. Lean blueprints expand every entropy/Ruzsa-distance/H[X]≥0 micro-lemma; this registry collapses foundational micro-lemmas into chapter-level theorem nodes.

Target: **~15–25 nodes per paper.** When the source paper has 50+ math environments, select the structurally important ones (theorems, propositions, key named lemmas) and let the foundational lemmas remain implicit in chapter-level summaries. Going to blueprint-fine granularity inflates node counts ~10× without adding navigation value.

## Honest reporting

- If a paper parses partially (math envs detected for some sections but not others), set `status: "partial"` in `index.json` rather than pretending complete.
- If a paper has 0 detected math environments (common in pre-2014 papers using non-standard `\newtheorem` macros), mark it explicitly rather than fabricating nodes.
- If uncertain about an edge direction, use a softer type (`addresses_same_question` over `depends_on`).
- When checking against a Lean blueprint, report what was *verified* (specific label matches) separately from what was *assumed* (structural correspondence).

The dataset is more valuable with 80% strong edges and a clear "we don't claim more" than with 100% confident edges where 20% are quietly wrong. See [`maynard-small-gaps/`](./maynard-small-gaps/) for the audit-pass standard.
